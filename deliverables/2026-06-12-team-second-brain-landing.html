How it works

A shared filing cabinet, two kinds of drawers

Every member runs the same MCP server pointed at one shared backend. One env var says who you are. The rest is just conversation.

01

One member creates the backend

A free Supabase project (recommended) or a shared git repo of markdown files. Paste-ready SQL from setup_bootstrap.

02

Each member sets their ID

Same config for everyone, except one line: MEMORYOS_OWNER_ID. That's who you are — it stamps your writes and scopes your reads.

03

Memory compounds

Writes default to private; say "shared" to publish to the team. Reads return your private records plus everything shared.

New in 0.8.0

Hardened mode: privacy the database enforces

Most team tools enforce privacy in the app layer and hope nobody looks underneath. Hardened mode closes that hole.

Per-member JWTs + Row Level Security

Even a teammate who bypasses the assistant and queries the database directly gets zero of your private rows.

Each member gets their own access token instead of a shared master key. Postgres Row Level Security checks the token's identity on every query — the database itself fails closed. Opt-in, fully backward-compatible, and verified by a live test suite that includes direct-to-database bypass probes.

SB_SUPABASE_ANON_KEY + SB_SUPABASE_MEMBER_JWT — that's the whole config change.
